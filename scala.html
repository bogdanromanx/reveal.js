Imperative

Describes computation in terms of statements that change the program state.

  • Modifying mutable variables
  • Use of assignments
  • Use of control structures (if-then-else, loops, break, continue, return)

C++ / Java / Objective-C / PHP / Scala etc.

Functional

Treats computations as an evaluation of mathematical functions and avoids state and mutable data.

  • No mutable variables
  • No assignments
  • No imperative control structures

Lisp / Closure / Haskell / F# / Scala etc.

Why Functional?

Managing state in multi-core is a nightmare

Functional programming techniques are inherently parallelizable

Allows compilers to perform optimizations

Provider higher level abstractions

Scala

  • General purpose language
  • Both Imperative and Functional
  • Compiles into Java bytecode and runs on the JVM
  • Interoperable with Java
  • Very strong static type system
  • Inspired by the criticism of Java shortcomings

Features of FP

  • First class / Higher order functions
  • Pure functions
  • Recursion - tail call optimizations
  • Unchecked exceptions
  • Non-strict (lazy) evaluation
  • Currying
  • Closures
  • Pattern matching
  • Type inference

Functions

A language is said to have first class functions if it treats functions as first class citizens.

A higher order function is a function that can take other functions as arguments or return functions.

Pure Functions

Pure functions have no side effects (memory / IO).

  • If the result of a pure expression is not used it can be removed.
  • The result of a pure function is constant with regards to its arguments (caching / memoization).
  • Pure functions with no data dependency are thread safe (can be executed in reversed order or in parallel).

Recursion

In FP iteration (looping) is accomplished via recursion.

Recursive functions invoke themselves until a base case is reached.

Tail call optimizations can be performed to avoid maintaining a stack.

Common patterns of recursion can be simplified by use of higher order functions (map / fold etc.).

Non-strict Evaluation

AKA: call-by-need / call-by-name

An evaluation strategy which delays the evaluation of an expression until its value is needed.

  • Can lead to smaller memory footprint
  • Can increase performance
  • Allows building infinite data structures (i.e.: streams)

Currying

A technique for partial function application.

Allows for a transformation of a function that takes multiple arguments in a way that can be called as a chain of functions.


                    f(x, y, z) => f(x)(y)(z)

Closures

A reference to a function together with its referencing environment.

Allows a function access to non-local variables when invoked outside its scope.

Pattern Matching

The act of perceiving the constituents of a data structure as constituents of a pattern.

Makes manipulations of complex data structures convenient and expressive.

(matching messaging in actor systems)
